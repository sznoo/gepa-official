# feedback_distance_v2: HotpotQA Retrieval-State Distance Diagnostics

This directory contains a controlled diagnostic pipeline for analyzing GEPA-style prompt feedback through **retrieval-state transitions** in a 2-hop HotpotQA BM25 setting.

The experiment is not a full non-oracle prompt-optimization pipeline. It is an **oracle-endpoint diagnostic** designed to test whether local retrieval transitions can be converted into useful sample-level prompt-update signals.

## 1. Research goal

The experiment studies whether GEPA-style prompt feedback can be interpreted through a **retrieval-state distance** view.

Core objects:

```text
q  = second-hop query generated by create_query_hop2
R  = retrieved documents / retrieval state after q
p  = prompt for create_query_hop2
R* = oracle-like target retrieval state based on gold support
```

The working hypothesis is:

```text
ΔR -> Δq -> Δp
```

The experiment separates two questions:

```text
R-state distance / midpoint trace:
  Can we construct metric-consistent intermediate retrieval states between current R and oracle-like R*?

Prompt-update absorption:
  Can local or full retrieval transitions be converted into an updated create_query_hop2 prompt that improves retrieval?
```

Main current conclusion:

```text
Intermediate retrieval traces are useful.
However, absorbing the full trace into one prompt update is too much.
A single local ΔR+Δq edge is stronger than endpoint-only update and full-trace aggregation under GPT-5-mini.
```

Qwen3-8B provides only weak diagnostic support:

```text
Qwen distance validity: PASS.
Qwen delta granularity: weak MR-side reproduction.
Qwen EM/downstream QA: unreliable due to reasoning-fragment query-generation artifacts.
```

## 3. HotpotQA program context

The underlying HotpotQA agent is the official two-hop BM25 program.

Relevant module structure:

```python
summarize1 = dspy.ChainOfThought("question,passages->summary")
create_query_hop2 = dspy.ChainOfThought("question,summary_1->query")
summarize2 = dspy.ChainOfThought("question,context,passages->summary")
final_answer = dspy.ChainOfThought("question,summary_1,summary_2->answer")
```

This experiment focuses on `create_query_hop2.predict`.

The main diagnostic target is the second-hop retrieval state:

```text
question + summary_1 -> create_query_hop2 -> q -> BM25 -> R -> summarize2 -> final_answer
```

### `scripts/run_rtrace_midpoint_validity_v2.py`

Purpose:

```text
Construct metric-consistent R-traces between current retrieval state and oracle target state.
```

Design:

```text
For each case:
  R_left  = current retrieval state or previous inserted midpoint
  R_right = oracle target state or previous right endpoint
  Generate R_mid candidate.
  Judge whether R_mid is valid relative to both endpoints:
    left_valid:  R_mid makes progress over R_left
    right_valid: R_mid remains meaningfully closer to / compatible with R_right
  If both valid, insert R_mid.
  If not valid, retry once with verifier feedback.
```

Outputs:

```bash
results/rtrace_midpoint_validity_v2_traces_hit48.jsonl
results/rtrace_midpoint_validity_v2_attempts_hit48.jsonl
results/rtrace_midpoint_validity_v2_summary_hit48.json
```

### `scripts/run_delta_granularity_prompt_update.py`

Purpose:

```text
Main prompt-update absorption experiment.
```

It tests whether a retrieval transition can be converted into a local update for the `create_query_hop2.predict` prompt.

Internal flow:

```text
generate_updated_prompt
  -> run_query_writer
  -> evaluate_query
  -> build_summary
```

More concretely:

```text
1. Build one or more delta observations from an R-trace.
2. Ask prompt updater to produce an updated create_query_hop2 prompt.
3. Run query writer under the updated prompt.
4. Search BM25 with the generated query.
5. Summarize retrieved docs using fixed summarize2 prompt.
6. Evaluate base and strong final answerers.
7. Compute retrieval metrics and EM.
```

Prompt updater system intent:

```text
Convert retrieval/query delta observations into a local update for a second-hop BM25 query-writer prompt.
The output is a prompt/instruction, not the query itself.
Prefer behavior-level prompt edits over literal entity memorization.
Make the smallest generalizable behavior change needed to improve the observed retrieval transition.
Return strict JSON with updated_prompt and rationale.
```

Supported arm sets:

```text
all
core
endpoint
single_rq
single_all
rq_only
```

Core arms:

```text
endpoint_R_only:
  input to prompt updater is coarse R0 -> R*
  query text is hidden

endpoint_Rq:
  input is coarse R0 -> R* plus q0 -> q*

full_trace_Rq:
  input is all adjacent ΔR_i + Δq_i along the constructed trace
```

Single arms:

```text
edge_i_Rq:
  input is one adjacent transition R_i -> R_{i+1} plus q_i -> q_{i+1}

best_single_edge_Rq:
  post-hoc best single edge per case, selected by missing_recovery_rate first,
  then strong_score, then arm_id
```

Important selection rule:

```python
sort key = -missing_recovery_rate, -strong_score, arm_id
```

Therefore `best_single_edge_Rq` is an MR-first oracle over edge choices, not an EM-first oracle.

## 11. Known pitfalls

### 11.1 Stale error rows

Earlier versions of `run_delta_granularity_prompt_update.py` produced stale error rows in JSONL files.

When summarizing manually, filter:

```python
rows = [r for r in rows if not r.get("error")]
```

### 11.2 Qwen `<think>` parser collapse

Invalid files:

```bash
results/delta_granularity_prompt_update_eval_genbias_single_rq_hit48_qwen.jsonl
results/delta_granularity_prompt_update_summary_genbias_single_rq_hit48_qwen.json
```

They parsed `<think>` as the query.

Use instead:

```bash
results/delta_granularity_prompt_update_eval_genbias_single_rq_hit48_qwen_reuseprompt_parsefix.jsonl
results/delta_granularity_prompt_update_summary_genbias_single_rq_hit48_qwen_reuseprompt_parsefix.json
results/delta_granularity_prompt_update_eval_genbias_core_hit48_qwen_reuseprompt_parsefix.jsonl
results/delta_granularity_prompt_update_summary_genbias_core_hit48_qwen_reuseprompt_parsefix.json
```

### 11.3 Qwen parsefix still not fully clean

Even after parsefix, many Qwen generated queries are reasoning fragments. Treat Qwen MR as weak diagnostic only.

### 11.4 Do not confuse MR and EM

Primary retrieval metric:

```text
missing_recovery_rate / MR
```

Answer-side metric:

```text
strong_EM
```

In Qwen runs, EM collapses and should not be used as controlled evidence.

### 11.5 Do not overinterpret oracle endpoints

`R*` is generated from `missing_after_hop2_titles + question`. It is a strong target state, not a true exhaustive oracle ceiling.
